## Usage

```bash
vulcn record [url] [options]
```

## Arguments

| Argument | Description                                     | Required |
| -------- | ----------------------------------------------- | -------- |
| `url`    | Target URL (overrides `target` in `.vulcn.yml`) | No       |

## Options

| Option                    | Description                                | Default        |
| ------------------------- | ------------------------------------------ | -------------- |
| `-o, --output <file>`     | Output filename (saved to `sessions/`)     | auto-generated |
| `-b, --browser <browser>` | Browser to use (chromium, firefox, webkit) | from config    |
| `--headless`              | Run in headless mode                       | `false`        |

## Description

The `record` command reads the target URL from `.vulcn.yml` (or a CLI argument), opens a browser, and captures your interactions. Every click, form input, and navigation is recorded into a session file that is saved to the `sessions/` directory. Press **Ctrl+C** to stop recording.

## How It Works

1. Reads target URL from `.vulcn.yml` (or CLI arg)
2. Opens the browser and records every interaction
3. Press **Ctrl+C** to stop and save to `sessions/`
4. Run `vulcn run` to replay with security payloads

## Examples

### Basic Recording

```bash
vulcn record
```

Opens the browser at the `target` URL from `.vulcn.yml`, records your interactions, and saves to `sessions/`.

### Override Target URL

```bash
vulcn record https://example.com
```

### Custom Output Filename

```bash
vulcn record -o login-flow.yml
```

This saves to `sessions/login-flow.yml`.

### Use Firefox

```bash
vulcn record -b firefox
```

## Session File Format

The recorded session is saved as a YAML file with the driver format:

```yaml
# sessions/recording-2026-02-14.yml
name: Session 2026-02-14
driver: browser
driverConfig:
  startUrl: https://example.com
  browser: chromium
steps:
  - id: step_001
    type: browser.navigate
    url: https://example.com

  - id: step_002
    type: browser.click
    selector: "#login-button"

  - id: step_003
    type: browser.input
    selector: "input[name='username']"
    value: testuser
    injectable: true

  - id: step_004
    type: browser.input
    selector: "input[name='password']"
    value: "********"
    injectable: true

  - id: step_005
    type: browser.click
    selector: "button[type='submit']"
```

## Step Types

| Type               | Description        | Fields              |
| ------------------ | ------------------ | ------------------- |
| `browser.navigate` | Page navigation    | `url`               |
| `browser.click`    | Mouse click        | `selector`          |
| `browser.input`    | Text input         | `selector`, `value` |
| `browser.select`   | Dropdown selection | `selector`, `value` |
| `browser.check`    | Checkbox toggle    | `selector`          |

Steps with `injectable: true` are targeted by the scanner for payload injection during `vulcn run`.

## Tips

<Tip>
  **Focus on input fields**: Record interactions that involve user input—forms,
  search boxes, comments. These are the injection points Vulcn will test.
</Tip>

<Tip>
  **Keep sessions focused**: Record one workflow per session (login, search,
  checkout). This makes it easier to identify which workflow triggered a
  vulnerability.
</Tip>

<Note>
  Vulcn automatically masks password fields in the session file. The actual
  values are not stored—only that a password field was filled.
</Note>
